
### 1) 2단계(메타 기반 RAG) 문제점 복기
1. top_k 중복 문제: 유사한 chunk가 같은 문서/인접 구간에서 반복되어 evidence 다양성이 떨어진다.  
2. 정렬 불일치: 벡터 유사도 순서가 질문에 직접 답하는 근거 순서와 다를 수 있다.  
3. 질의 품질 흔들림: 구어체/장문 질문은 검색에 불리해 recall/precision이 불안정해진다.  
4. 후보 부족: top_k가 작으면 좋은 근거가 있어도 후보에 못 들어와 놓칠 수 있다.

---

### 2) 해결 전략(3단계)
Retrieval 파이프라인을 다음처럼 업그레이드한다.

1) Query rewrite  
- 사용자 질문 → 검색 친화형(키워드/제약 중심) 짧은 질의로 정리

2) fetch_k 후보 확장  
- top_k보다 훨씬 많이 후보를 확보(예: 20~60) 후 선별

3) MMR 선택
- 후보 중 중복을 줄이고, 서로 다른 근거를 top_k로 선택

4) LLM Re-rank  
- 질문에 직접 답하는 근거 기준으로 evidence 순서를 재정렬

데이터 구조는 2단계 그대로 유지한다.
- chunks = [{"text": "...", "meta": {..., "chunk_id": n}}]

3단계에서 retrieve() 반환이 바뀐다.
- search_query, hits = retrieve(...)

UI에서 검색용 질의를 표시해 “왜 이 근거가 나왔는지”를 투명하게 한다.
- ex) 검색용 질의(rewrite): ...

### 3) 구성 요소 설명(핵심)
#### (A) Query rewrite
- 목적: 검색에 유리한 형태로 질문을 정리(핵심 키워드/제약 유지)
- 효과: 애매한 질문에서 recall/precision 안정화

#### (B) fetch_k
- 목적: top_k만으로는 부족하므로 후보를 넉넉히 확보해 선택의 여지를 만든다.
- 권장: fetch_k = max(top_k*4, 20)

#### (C) MMR
- 목적: 비슷한 chunk 반복을 줄이고 다양한 근거를 선택한다.
- 파라미터:
  - lambda → 1.0: 관련성 우선(중복 증가 가능)
  - lambda → 0.0: 다양성 우선(관련성 흔들릴 수 있음)

#### (D) Re-rank
- 목적: 벡터 점수 순서 대신 직접 답 근거를 위로 올린다.
- 방법: LLM이 후보를 읽고 순서만 JSON으로 반환(답 생성 X)

---

### 4) 실습 시나리오(권장 진행 순서)
1) baseline* rewrite OFF, MMR OFF, rerank OFF  
2) rewrite만 ON: 질의 변화와 검색 결과 변화 관찰  
3) fetch_k 증가 + MMR ON: evidence 중복 감소/다양성 증가 확인  
4) rerank ON: 답에 직접 필요한 근거가 앞 evidence로 올라오는지 확인
